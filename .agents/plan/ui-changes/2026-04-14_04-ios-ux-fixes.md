## Bug 1 — Settings Language Buttons Render Faded

**Symptom:** Language buttons appear washed out on first render, normalize after any interaction.

**Root Cause Analysis:**
The settings screen builds language buttons inline (lines 252-278 of `settings.tsx`). Colors come from `useTheme()` → `useSettingsStore(s => s.sex)`. If theme colors resolve correctly, the issue is likely **iOS-specific rendering** with 8-digit hex colors (`withOpacity(colors.primary, '10')`). iOS sometimes delays rendering 8-digit hex backgrounds, causing text to briefly inherit a faded appearance.

Alternative: the `useFocusEffect` async block (lines 80-92) triggers a state update that causes a re-render, but the initial render with the first paint may use stale opacity values.

**Fix Strategy:**
- Replace `withOpacity(colors.primary, '10')` for active language button background with an `rgba()` equivalent using a helper
- OR: Add `opacity` style property instead of 8-digit hex alpha channel
- Test: confirm buttons render correctly on first paint

**Affected files:** `app/(tabs)/settings.tsx`

## Bug 2 — Date Picker Broken in EditProfileModal (iOS) / Double-Tap (Android)

**Symptom (iOS):** Tapping "date" button does nothing. After attempting, other settings controls stop responding.

**Symptom (Android):** Sometimes need to tap date button twice to open picker.

**Root Cause Analysis:**

**iOS — Stacked Modals:**
`EditProfileModal` renders as a transparent `<Modal>`. Inside it, `WheelDatePickerModal` renders as another `<Modal>`. On iOS, stacked native Modals cause touch-event routing issues — the second modal can steal the responder tree, and after dismissal, the first modal's touch handlers may not recover.

**Android — Keyboard Dismiss Steals First Tap:**
`EditProfileModal` backdrop has `onPress={() => { Keyboard.dismiss(); dismissModal(); }}`. If the keyboard is open and user taps the date button, the first tap may be consumed by `keyboardShouldPersistTaps` behavior or RN's keyboard dismiss logic.

**Fix Strategy (iOS):**
- **Option A (recommended):** Replace `WheelDatePickerModal`'s outer `<Modal>` with a conditional `View` overlay rendered inside `EditProfileModal`. This avoids nested native Modals entirely.
- **Option B:** Use `presentationStyle="overFullScreen"` on the inner modal to keep the outer modal's responder tree alive.

**Fix Strategy (Android):**
- The `DatePickerField` (used by `AddWordModal`) already has `keyboardShouldPersistTaps="handled"` via `KeyboardAwareScrollView`. For `EditProfileModal`, the same is true (line 104-106). The issue is that `EditProfileModal`'s date button is a `TouchableOpacity` that calls `setShowPicker(true)`. When the keyboard is up, we should call `Keyboard.dismiss()` inside `openPicker` before `setOpen(true)`, or add a small delay.

**Affected files:** `src/components/EditProfileModal.tsx`, `src/components/WheelDatePickerModal.tsx`

## Bug 3 — Keepsake Preview Too Tall for iPhone 16/17

**Current dimensions:**
- `PREVIEW_SCALE = 391 / 1080` ≈ 0.362
- Preview wrapper: 391dp × 694dp
- iPhone 16/17 screen: 393dp × 852dp
- Visible area after header (72dp) + actions (~80dp) + safe area (~48dp) = ~652dp → needs scrolling

**Fix:**
- Reduce `PREVIEW_SCALE` by 10%: `352 / 1080` ≈ 0.326
- Update `previewWrapper` to width `352`, height `352 * (1920 / 1080)` ≈ 626dp
- This fits within ~652dp available

**Design tokens changed:**
- `PREVIEW_SCALE` constant: `391/1080` → `352/1080`
- `previewWrapper` width: `391` → `352`

**Affected files:** `src/components/keepsake/KeepsakePreviewModal.tsx`

## Bug 5 — AddWordModal Starts Scrolled to Bottom on iOS

**Symptom:** Modal content starts at the bottom (near action buttons), then scrolls up to show the word input after ~1 second.

**Root Cause Analysis:**
`KeyboardAwareScrollView` from `react-native-keyboard-controller` may compute content offset based on `autoFocus` on the word input. On iOS, the keyboard opens immediately when `autoFocus` fires, and the scroll view adjusts position to keep the focused input visible — but it overshoots to the bottom first because the full content hasn't been measured yet.

**Fix Strategy:**
- Remove `autoFocus` from word input when creating new words, and instead use `useEffect` with `inputRef.current?.focus()` after a short delay (100-200ms) to let the modal settle
- OR: add `contentContainerStyle={{ flexGrow: 1 }}` to prevent over-scroll, combined with `enableOnAndroid={false}` since Android handles keyboard avoidance natively
- OR: wrap focus trigger with `InteractionManager.runAfterInteractions()` so focus happens after modal animation completes

**Affected files:** `src/components/AddWordModal.tsx`
