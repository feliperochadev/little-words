## Refined Prompt

### 1. Context & Motivation
Five iOS-specific UX bugs discovered during iPhone 16/17 testing. All are visual or interaction regressions that degrade the iPhone experience. Fixes are targeted — no architectural changes.

### 2. Precise Requirements

**Bug 1 — Settings Language buttons render faded on iOS initial load**
- Settings screen language buttons appear washed out (text barely visible) on first render
- Returns to normal after interacting with another component or screen change
- Root cause hypothesis: stale theme colors on initial mount, or iOS-specific rendering quirk with 8-digit hex opacity
- Fix: ensure language buttons use stable colors on first paint; may need to force re-render or switch opacity strategy

**Bug 2 — EditProfileModal date picker broken on iOS; double-tap required on Android**
- On iOS: tapping date button in EditProfileModal does nothing; subsequent taps on other settings controls also fail (modal blocks interaction)
- On Android: DatePickerField sometimes requires two taps to open
- Root cause hypothesis (iOS): stacked transparent Modals (EditProfileModal → WheelDatePickerModal) cause iOS touch event issues
- Root cause hypothesis (Android): first tap dismisses keyboard (via `keyboardShouldPersistTaps` or `Keyboard.dismiss()` in backdrop), second tap actually fires
- Fix (iOS): replace WheelDatePickerModal's outer `<Modal>` with conditional rendering inside EditProfileModal, or use `presentationStyle` fix
- Fix (Android): ensure `keyboardShouldPersistTaps="handled"` propagates, or dismiss keyboard before opening picker

**Bug 3 — Keepsake preview card too tall for iPhone 16/17**
- Current: `PREVIEW_SCALE = 391 / 1080` → 391dp wide, ~694dp tall
- iPhone 16/17 visible height after header/actions is ~600-650dp, requires scrolling
- Fix: reduce `PREVIEW_SCALE` by 10% → `352 / 1080` (~0.326), update `previewWrapper` dimensions to match

**Bug 4 — Keepsake preview action buttons font too large for Portuguese**
- Current: `actionBtnText` fontSize 15
- Portuguese "Salvar no dispositivo" overflows the button
- Fix: reduce fontSize by 15% → round to 13; adjust both save and share buttons

**Bug 5 — AddWordModal starts scrolled to bottom on iOS keyboard open**
- `KeyboardAwareScrollView` from `react-native-keyboard-controller` scrolls to end of modal content initially, then adjusts upward after ~1s
- Expected: modal should start at top showing the word text input
- Fix: investigate `KeyboardAwareScrollView` initial scroll behavior; may need to set `scrollToOverflowEnabled={false}` or add initial `scrollTo({y: 0})`, or configure `keyboardOffset`
